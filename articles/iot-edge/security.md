# <a name="security-standards-for-azure-iot-edge"></a>適用於 Azure IoT Edge 的安全性標準

Azure IoT Edge 解決了將資料和分析移動到智慧邊緣時固有的風險。 IoT Edge 安全標準將不同部署方案的靈活性與所有 Azure 服務所需的保護平衡。

IoT Edge 在各種硬體型號和型號上運行，支援多個作業系統，並適用于不同的部署方案。 IoT Edge 不是為特定方案提供具體解決方案，而是基於為規模設計的有良好基礎的原則的可擴展安全框架。 部署方案的風險取決於許多因素，包括：

* 解決方案擁有權
* 部署地理
* 資料靈敏度
* 隱私權
* 垂直應用
* 監管要求

本文會概述 IoT Edge 的安全性架構。 如需詳細資訊，請參閱[保護智慧邊緣](https://azure.microsoft.com/blog/securing-the-intelligent-edge/) \(英文\)。

## <a name="standards"></a>標準

標準會提昇安全性和實作的簡便性，兩者都是安全性的特點。 安全性解決方案應該投入安全性的評估，來建置信任且不成為部署的障礙。 用來保護 Azure IoT Edge 的架構在設計時，是依據經過時間考驗及業界證實的安全性通訊協定，因此具有熟悉和重複使用特性。

## <a name="authentication"></a>驗證

部署 IoT 解決方案時，您需要知道只有受信任的參與者、設備和模組才能訪問您的解決方案。 基於證書的身份驗證是 Azure IoT 邊緣平臺身份驗證的主要機制。 此機制源自互聯網工程任務組 （IETF） 管理公開金鑰基礎結構 （PKiX） 的一組標準。

與 Azure IoT Edge 設備交互的所有設備、模組和參與者都應具有唯一的證書標識。 本指南適用于交互是物理的還是通過網路連接的。 並不是每個方案或元件都適合基於證書的身份驗證，因此安全框架的可擴充性提供了安全的替代方案。

有關詳細資訊，請參閱[Azure IoT 邊緣證書使用方式](iot-edge-certs.md)。

## <a name="authorization"></a>授權

最低權限原則的含意是，系統的使用者和元件只應具有執行其角色所需最小一組資源和資料的存取權。 裝置、模組和執行者只應存取其權限範圍內的資源和資料，且只有在架構上允許時才能存取。 某些許可權可配置，具有足夠的許可權，而其他許可權則通過體系結構強制執行。 例如，某些模組可能授權連接到 Azure IoT 中心。 但是，沒有理由讓一個 IoT Edge 設備中的模組訪問另一個 IoT Edge 設備中的模組孿生。

其他授權方案包括證書簽名許可權和基於角色的存取控制 （RBAC）。

## <a name="attestation"></a>證明

認證可確保軟體位的完整性，這對檢測和預防惡意軟體非常重要。 Azure IoT Edge 安全性架構會將證明分類在三個主要類別之下：

* 靜態證明
* 執行階段證明
* 軟體證明

### <a name="static-attestation"></a>靜態證明

靜態驗證在通電期間驗證設備上所有軟體的完整性，包括作業系統、所有運行時和配置資訊。 由於靜態證明在通電期間發生，因此通常稱為安全啟動。 IoT Edge 設備的安全框架擴展到製造商，並集成了安全硬體功能，可確保靜態認證過程。 這些程序包括安全開機和安全韌體升級。 與晶片廠商密切合作可以排除非必要的韌體層，藉此將威脅表面降至最低。

### <a name="runtime-attestation"></a>執行階段證明

一旦系統完成了安全啟動過程，精心設計的系統應檢測注入惡意軟體的嘗試並採取正確的對策。 惡意軟體攻擊可能針對系統的埠和介面。 如果惡意參與者對設備具有物理存取權限，則他們可能會篡改設備本身或使用側通道攻擊來獲取存取權限。 此類惡意內容（無論是惡意軟體還是未經授權的配置更改）無法通過靜態證明來檢測，因為它是在啟動過程後注入的。 裝置硬體所提供或強制執行的因應對策有助於閃避這類威脅。 IoT Edge 的安全框架明確要求擴展來打擊運行時威脅。  

### <a name="software-attestation"></a>軟體證明

所有健康系統（包括智慧邊緣系統）都需要修補程式和升級。 安全性對於更新過程非常重要，否則它們可能是潛在的威脅媒介。 IoT Edge 的安全框架通過測量和簽名的包要求更新，以確保包源的完整性並對其進行身份驗證。 此一標準適用於所有作業系統和應用程式軟體位元。

## <a name="hardware-root-of-trust"></a>硬體信任根源

對於許多智慧邊緣設備，尤其是潛在惡意參與者可以物理訪問的設備，硬體安全是保護的最後一道防線。 可防竄改的硬體對於這類部署非常重要。 Azure IoT Edge 股利安全晶片硬體廠商提供不同類別的硬體信任根源，以因應各種風險設定檔和部署案例。 硬體信任可能源自常見的安全性通訊協定標準，例如信賴平台模組 (ISO/IEC 11889) 和信賴運算群組的 Device Identifier Composition Engine (DICE)。 安全保護區技術 (例如，TrustZones 和 Software Guard Extensions (SGX)) 也能提供硬體信任。

## <a name="certification"></a>認證

為了説明客戶在採購 Azure IoT Edge 設備進行部署時做出明智的決策，IoT Edge 框架包括認證要求。 這些需求的基本是有關於安全性宣告的憑證，和有關於安全性實作驗證的憑證。 例如，安全聲明認證意味著 IoT Edge 設備使用已知用於抵抗引導攻擊的安全硬體。 驗證認證意味著安全硬體已正確實現，以在設備中提供此值。 為求簡單，此架構會嘗試讓憑證保持最低的負擔。

## <a name="extensibility"></a>擴充性

隨著 IoT 技術推動不同類型的業務轉型，安全性應並行發展，以解決新出現的方案。 Azure IoT Edge 安全性架構是從穩固基礎開始，在這個基礎上建置擴充性到不同的維度以納入：

* 第一方安全性服務，例如 Azure IoT 中樞的裝置佈建服務。
* 協力廠商服務，如為不同應用垂直（如工業或醫療保健）或技術焦點（如網狀網路中的安全監控或矽硬體證明服務）提供託管安全服務，合作 夥伴。
* 舊版系統，以包含具有其他安全性策略的更動，例如使用憑證以外的安全技術來進行驗證和身分識別管理。
* 安全硬體，以便採用新興安全硬體技術和晶片合作夥伴貢獻。

最後，確保智慧優勢需要來自一個開放社區的協作貢獻，這些社區由保護 IoT 的共同利益驅動。 這些貢獻的形式可能是安全技術或服務。 Azure IoT Edge 安全性架構提供安全性的穩固基礎，可以針對涵蓋範圍最大值加以擴充，在智慧型邊緣中提供與 Azure 雲端相同等級的信任和完整性。
