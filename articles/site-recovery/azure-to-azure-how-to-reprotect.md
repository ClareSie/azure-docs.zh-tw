# <a name="reprotect-failed-over-azure-vms-to-the-primary-region"></a>重新保護已容錯移轉到主要區域的 Azure VM

使用[Azure 網站恢復](site-recovery-overview.md)將 Azure VM 從一個區域[容錯移轉到](site-recovery-failover.md)另一個區域時，VM 將啟動到次要區域，處於**未受保護的**狀態。 如果要將 VM 故障退回主區域，執行以下任務：

1. 重新保護次要區域中的 VM，好讓它們開始複寫到主要區域。
1. 重新保護完成後，VM 正在複製，則可以從次要區域容錯移轉到主區域。

## <a name="prerequisites"></a>Prerequisites

- 必須認可從主要到次要區域的 VM 容錯移轉。
- 主要目標網站應該可供使用，而且您應該能夠存取或建立該區域中的資源。

## <a name="reprotect-a-vm"></a>重新保護 VM

1. 在**Vault** > **複製專案中**，按右鍵 VM 上的失敗，然後選擇 **"重新保護**"。 重新保護的顯示方向應該是從次要到主要。

   ![重新保護](./media/site-recovery-how-to-reprotect-azure-to-azure/reprotect.png)

1. 檢閱資源群組、網路、儲存體和可用性設定組。 然後按一下 **[確定]**。 如果有任何資源標記為新資源，則這些資源將作為重新保護過程的一部分創建。
1. 重新保護作業會在目標網站植入最新資料。 作業完成後，將進行增量複製。 然後，您可以容錯移轉回到主要網站。 您可以使用自訂選項，來選取您想要在重新保護期間使用的儲存體帳戶或網路。

   ![自訂選項](./media/site-recovery-how-to-reprotect-azure-to-azure/customize.png)

### <a name="customize-reprotect-settings"></a>自訂重新保護設定

您可以在重新保護期間自訂目標 VM 的以下屬性。

![自訂](./media/site-recovery-how-to-reprotect-azure-to-azure/customizeblade.png)

|屬性 |注意  |
|---------|---------|
|目標資源群組 | 修改要在其中建立 VM 的目標資源群組。 目標 VM 會在重新保護過程中加以刪除。 您可以選擇容錯移轉之後要在其中建立 VM 的新資源群組。 |
|目標虛擬網路 | 目標網路無法在重新保護作業期間進行變更。 若要變更網路，請重新進行網路對應。 |
|目標存儲（輔助 VM 不使用託管磁片） | 您可以變更 VM 在容錯移轉之後所使用的儲存體帳戶。 |
|複本受控磁碟 (次要 VM 使用受控磁碟) | 站台復原會在主要區域中建立複本受控磁碟，以便製作次要 VM 受控磁碟的鏡像。 |
|快取儲存體 | 您可以指定要在複寫期間使用的快取儲存體帳戶。 預設情況下，如果不存在，則會創建新的緩存存儲帳戶。 |
|可用性設定組 | 如果次要區域中的 VM 是可用性設定組的一部分，您就可以針對主要區域中的目標 VM 選擇可用性設定組。 根據預設，Site Recovery 會在主要區域中嘗試尋找現有的可用性設定組並加以使用。 進行自訂時，您可以指定新的可用性設定組。 |

### <a name="what-happens-during-reprotection"></a>重新保護期間會發生什麼情況？

預設情況下，會發生以下情況：

1. 在容錯移轉之 VM 的執行區域中建立一個快取儲存體帳戶。
1. 如果目標儲存體帳戶 (主要區域中的原始儲存體帳戶) 不存在，即會建立新的帳戶。 分配的存儲帳戶名稱是輔助 VM 使用的存儲帳戶的名稱，尾碼為`asr`。
1. 如果 VM 使用受控磁碟，即會在主要區域中建立複本受控磁碟，以儲存從次要 VM 磁碟複寫的資料。
1. 如果目標可用性集不存在，則在必要時作為重新保護作業的一部分創建一個新可用性集。 如果已自訂重新保護設置，則使用所選集。

當您觸發重新保護作業且目標 VM 存在時，會發生下列情況：

1. 如果目標端 VM 正在執行，請加以關閉。
1. 如果 VM 使用託管磁片，則使用`-ASRReplica`尾碼創建原始磁片的副本。 原始磁碟則會被刪除。 副本`-ASRReplica`用於複製。
1. 如果 VM 使用非受控磁碟，就會將目標 VM 的資料磁碟中斷連結並用於複寫。 將會建立 OS 磁碟的複本並連結至 VM。 原始 OS 磁碟則會中斷連結並用於複寫。
1. 只會針對來源磁碟與目標磁碟之間的變更進行同步處理。 系統會藉由比較這兩個磁碟來計算出差異，然後進行傳輸。 檢查下面，查找完成重新保護的估計時間。
1. 同步完成後，增量複製將開始，並且根據複寫原則創建復原點。

當您觸發重新保護作業，並且目標 VM 和磁片不存在時，將發生以下情況：

1. 如果 VM 使用託管磁片，則使用`-ASRReplica`尾碼棄置站台磁片。 副本`-ASRReplica`用於複製。
1. 如果 VM 使用非受控磁碟，就會在目標儲存體帳戶中建立複本磁碟。
1. 將會把全部磁碟從容錯移轉的區域複製到新的目標區域。
1. 同步完成後，增量複製將開始，並且根據複寫原則創建復原點。

#### <a name="estimated-time-to-do-the-reprotection"></a>完成重新保護的估計時間

在大多數情況下，Azure 網站恢復不會將完整資料複製到源區域。
以下條件確定複製的資料量：

1. 如果源 VM 資料由於某種原因（如資源組更改/刪除）被刪除、損壞或無法訪問，則在重新保護期間將發生完整的初始複製，因為源區域上沒有可用的資料可供使用。
1. 如果源 VM 資料是可訪問的，則僅通過比較磁片然後傳輸來計算差異。 檢查下表以獲取估計時間。

|示例情況 | 重新保護所需要時間 |
|---|---|
|源區域具有 1 個 VM，具有 1 個 TB 標準磁片。<br/>僅使用 127 GB 資料，其餘磁片為空。<br/>磁片類型是標準磁片類型，輸送量為 60 MiB/S。<br/>容錯移轉後無資料更改。| 大致時間：45 分鐘 = 1.5 小時。<br/>在重新保護期間，網站恢復將填充所有資料的檢查和，這些資料需要 127 GB/45 MB，大約 45 分鐘。<br/>網站恢復自動縮放需要一些開銷時間，大約 20-30 分鐘。<br/>無出口費用。 |
|源區域具有 1 個 VM，具有 1 個 TB 標準磁片。<br/>僅使用 127 GB 資料，其餘磁片為空。<br/>磁片類型是標準磁片類型，輸送量為 60 MiB/S。<br/>容錯移轉後 45 GB 資料更改。| 大致時間：1 小時 = 2 小時。<br/>在重新保護期間，網站恢復將填充所有資料的檢查和，這些資料需要 127 GB/45 MB，大約 45 分鐘。<br/>傳輸時間以應用 45 GB 的更改，即 45 GB/45 MBps，大約 17 分鐘。<br/>出口費用將用於 45 GB 資料更改，而不是校驗和。 |

當 VM 在故障返回主區域（即，如果 VM 從主區域重新保護到 DR 區域）後重新受到保護時，將刪除目標 VM 和關聯的 NIC（s）。

當 VM 從 DR 區域重新保護到主區域時，我們不會刪除前主 VM 和關聯的 NIC。

## <a name="next-steps"></a>後續步驟

當 VM 受到保護之後，您就能起始容錯移轉。 容錯移轉將關閉次要區域中的 VM，並在主區域中創建和引導 VM，在此過程中會短暫停機。 我們建議您為此過程選擇適當的時間，並在啟動對主網站的完全容錯移轉之前運行測試容錯移轉。 [瞭解有關](site-recovery-failover.md)Azure 網站恢復容錯移轉的更多詳細資訊。
