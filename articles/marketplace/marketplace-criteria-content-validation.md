# <a name="azure-marketplace-listing-guidelines"></a>Azure Marketplace 列名指導方針

本文件包含適用於檢閱者針對列名至 Azure Marketplace 之供應項目和服務的需求指導方針與檢查清單。

所有供應項目都必須符合以下所示的[所有列名需求](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#all-listing-requirements)。 針對特定列名類型，會提供其他需求和檢查清單，包括：附加[試用版](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#trial-offer-requirements)的供應項目、[SaaS 應用程式](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#saas-app-requirements)、[容器](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#container-offer-requirements)和[諮詢供應項目](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#consulting-offer-requirements)。


## <a name="all-listing-requirements"></a>所有列名需求

|  | 列名項目 | 基本需求 | 最佳需求 |
|:--- |:--- |:--- |:--- |
| 1 | 潛在客戶目的地 | 已設定潛在客戶目的地| 具有潛在客戶目的地 CRM 資訊的 OCP 目錄也列在合作夥伴解決方案索引標籤中 | 
| 2 | 供應項目標題 | 描述解決方案供應項目。 在合作夥伴的網站上匹配解決方案的線上推廣 | 包含搜尋關鍵字 |
| 3 | 標誌 | 正確地顯示標誌 | 徽標顯示正確，並在 Azure 門戶中包含 [英雄圖像，這是大格式圖像 |
| 4 |  供應項目描述 | <br>• 2 到 3 個段落 <br>• 一目了然的解決方案供應項目 <br>• 供應項目描述沒有任何拼字和文法錯誤 <br>• 優惠描述是全面的和捕獲的：目標受眾，使用者類型，為什麼它的價值（價值道具） <br>• 供應項目描述採用段落敘述形式搭配容易理解的簡短句子 | <br>• 目標產業概述 (如果有相關) <br>• 良好的樣式格式，每個段落標題都有一個句子或片語概述後續的內容，且包含項目符號 (如果適當時) 以強調重要的優點。 此處的目標是讓閱讀者能以易於檢視的格式一眼就了解供應項目，而不需要閱讀段落內容。 <br>• 每個段落之間都有間距，且讀起來像汽車手冊。 意思並不是要描述技術功能或程式碼術語，而是以簡單的字詞和完整的描述來說明供應項目。 | 
| 5 | 類別和產業 | <br>• 類別與解決方案供應項目功能相符 <br>• 不會延伸至與解決方案不符的類別 <br>• 最適合的產業，或所有選取的產業 (如果沒有最適合的產業) | <br>• 最多 3 個類別 <br>• 最多 3 個產業 |
| 6 | 影像 | <br>• 影像並非必要，但如果有提供，則必須顯示且為高解析度影像。 <br>• 影像需求列在 CPP 中以及此處 <br>• 包含在螢幕擷取畫面中的文字是清楚易於閱讀的影像 | 解決方案供應項目是一目了然的 |
| 7 | 影片 | <br>• 影片並非必要，但有提供，則必須能播放而不會發生任何錯誤。 <br>• 如果有提供，不得提及競爭對手公司，「除非」是示範移轉解決方案 |<br>• 理想長度為 3 分鐘以上 <br>• 可透過影片內容輕鬆了解解決方案供應項目 <br>• 解決方案功能的示範 |
| 8 | 列名狀態 (行動呼籲) | <br>必須標記以下類型之一： <ul>o 與我聯絡</ul><ul>o 試用版/立即取得試用版/開始試用/試用產品</ul><ul>o 立即購買/立即購買</ul> | 客戶可以輕易地了解有哪些可用的後續步驟： <ul>1. 試用試用版</ul> <ul>2. 立即購買</ul><ul>3. 通過電子郵件或電話號碼聯繫，安排 PoC、評估或簡報。</ul> | 
| 9 | 解決方案價格 | 必須有解決方案價格索引標籤/詳細資料，且以合作夥伴解決方案供應項目的當地貨幣為單位 | 使用定價層提供多種計費選項供客戶選擇 |    
| 10 | 深入了解 | 底部的連結（在描述下，而不是左側的市場連結）導致有關解決方案的詳細資訊，並且公開可用並正確顯示 |    指向特定專案的連結（例如，合作夥伴網站上的規格頁面），而不僅僅是合作夥伴主頁網站 | 
| 11 | 解決方案支援與說明 | 提供至少下列一個的連結： <br>o   電話號碼 <br>o 電子郵件支援 <br>o 聊天代理程式 <br>o   社群論壇 |<br>• 列出所有支援方法：電話與電子郵件支援。 <br>• 在試用版或試用產品期間提供免費的付費支援 | 
| 12 | 法律 | 透過公開 URL 提供原則或條款 |  |

## <a name="consulting-offer-requirements"></a>諮詢供應項目需求

|  | 列名項目 | 基本需求 | 最佳需求 |
|:--- |:--- |:--- |:--- |
| 1 | 供應項目標題  |  <br>• 必須以以下格式明確列出服務類型和持續時間：名稱：持續時間類型。 （即"優惠參與：為期 1 周的概念證明"）  | <br>• 不重複的發行者名稱 <br>• 描述中已提供鏡像標題樣式，不需要長標題。 | 
| 2 | 供應項目描述  |<br>• 確定適當地使用 [Microsoft 產品名稱](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/en-us.aspx) <br>• 供應項目註明價格：估價必須在供應項目的頂端或底部註記以說明變化性 (到用戶端出差、要移轉的伺服器數量等等) <br>• 每個供應項目都有描述需求，如下所示： <ul><br>o 簡報中針對簡報涵蓋的主題資訊至少需要 4 到 5 個項目符號來說明 </ul> <ul><br>o 所有工作坊都需要日程表 </ul><ul><br>o 所有日程表都必須細分以日或週為單位，視工作坊的持續時間而定 </ul><ul><br>o 評量、POC、非訓練型工作坊，實作的供應項目則需要交付項目 </ul><ul><br>o 培訓研討會不需要可交付成果，但它們需要更詳細的議程，其中將涵蓋主題。 | <br>• 任何供應項目都有日程表和交付項目 <br>• 供應項目頂端區段包含一個段落，含有公司提供之服務的相關內容 <br>• 供應項目包含本身服務之價值的段落，作為頂端區段 |
| 3 | Markdown 格式  | 所有供應項目都必須使用 Markdown 格式，如此在轉換為 HTML 時才會正確轉譯供應項目  |  |
| 4 | 類別和產業 | 不相關的類別 |  | 
| 5 | 列名狀態 (CTA) | 自動列為與我連絡 |  | 
| 6 | 解決方案支援與說明  | 支援與說明為非必要  | | 
| 7 | 提供隱私權原則及使用條款  |  原則或條款為非必要  | | 
| 8 | 服務類型 | 確定服務類型與標題相符 | |
| 9 | 專長認證 |<br>• 必須至少為下列其中一個：<br>• 應用程式開發 <br>• 應用程式整合 <br>• 應用程式生命週期管理 <br>• 雲端平台 <br>• 資料分析 <br>• 資料中心 <br>• 資料平台 <br>• DevOps |
| 10 | Products | 必須是 Azure | |
| 11 | 國家/區域 | 確定國家和地區與選擇的貨幣相符 | |
| 12 | 深入了解  | <br>• 底部的連結（在描述下，而不是左側的市場連結）可獲取有關解決方案的詳細資訊，並公開提供並正確顯示。 <br>• 連結必須具有"易記名稱"，不得顯示為任何下載的檔案名 | |
