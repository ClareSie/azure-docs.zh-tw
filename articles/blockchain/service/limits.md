# <a name="limits-in-azure-blockchain-service"></a>Azure 區塊鏈服務中的限制

Azure 區塊鏈服務具有服務和功能限制，例如成員可以擁有的節點數目、聯盟限制和儲存體數量。

## <a name="pricing-tier"></a>定價層

交易和驗證程式節點的上限取決於您是在基本或標準定價層布建 Azure 區塊鏈服務。

| 定價層 | 最大交易節點 | 最大驗證程式節點 |
|:---|:---:|:---:|
| 基本 | 10 | 1 |
| 標準 | 10 | 2 |

您的聯盟網路應該要有至少兩個 Azure 區塊鏈服務標準層節點。 標準層節點包含兩個驗證程式節點。 需要四個驗證程式節點，才能符合 [伊斯坦布爾的拜占庭容錯共識](https://github.com/jpmorganchase/quorum/wiki/Quorum-Consensus)。

使用「基本」層級適用于開發、測試和概念證明。 針對生產等級部署使用標準層。 如果您使用區塊鏈資料管理員或傳送大量的私人交易，也應該使用*標準*層。

不支援在建立成員之後變更基本和標準間的定價層。

## <a name="storage-capacity"></a>儲存體容量

針對總帳資料和記錄，每個節點可以使用的儲存體數量上限為 1.8 tb。

不支援減少總帳和記錄儲存體大小。
## <a name="consortium-limits"></a>聯盟限制

* **聯盟和成員名稱必須是** Azure 區塊鏈服務中其他聯盟和成員名稱的唯一名稱。

* **無法變更成員和聯盟名稱**

* **聯盟中的所有成員都必須位於相同的定價層**

* **參與聯盟的所有成員都必須位於相同區域**

    聯盟中所建立的第一個成員會規定區域。 聯盟的受邀成員必須位於與第一個成員相同的區域中。 將所有成員限制在相同區域，有助於確保網路共識不會受到負面影響。

* **聯盟至少必須有一個系統管理員**

    如果聯盟中只有一個系統管理員，他們就無法從聯盟移除自己或刪除其成員，除非在聯盟中新增或升級其他系統管理員。

* **無法再次新增從聯盟移除的成員**

    相反地，他們必須 reinvited 來加入聯盟，然後建立新的成員。 不會刪除其現有的成員資源來保留歷程記錄交易。

* **聯盟中的所有成員都必須使用相同的總帳版本**

    如需 Azure 區塊鏈服務中可用之修補、更新和總帳版本的詳細資訊，請參閱 [修補、更新及版本](ledger-versions.md)。

## <a name="performance"></a>效能

請勿使用 *eth。預估* 每個提交交易的天然氣功能。 *Eth. 估計*函數是記憶體密集型。 多次呼叫函式會大幅減少每秒的交易數。

可能的話，請使用保守的天然氣價值來提交交易，並儘量減少使用 *eth。估價*。
