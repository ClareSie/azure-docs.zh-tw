# <a name="configure-the-admin-consent-workflow-preview"></a>建構管理員同意工作流（預覽）

本文介紹如何啟用管理員同意工作流（預覽）功能，該功能為最終使用者提供了一種請求訪問需要管理員同意的應用程式的方法。

如果沒有管理員同意工作流，當租戶嘗試訪問任何需要訪問組織資料許可權的應用時，將阻止使用者同意被禁用。 使用者看到一條通用錯誤訊息，指出他們未經授權訪問應用，他們應該向管理員尋求説明。 但是，使用者通常不知道要聯繫誰，因此他們要麼放棄，要麼在應用程式中創建新的本地帳戶。 即使收到通知，也並不總是有簡化的過程來説明管理員授予存取權限並通知其使用者。
 
管理員同意工作流為管理員提供了一種安全的方式來授予對需要管理員批准的應用程式的存取權限。 當使用者嘗試訪問應用程式但無法提供同意時，他們可以發送管理員批准請求。 請求通過電子郵件發送給被指定為檢閱者的管理員。 檢閱者對請求執行操作，並通知使用者該操作。

要批准請求，檢閱者必須是全域管理員、雲應用程式管理員或應用程式管理員。 檢閱者必須已經分配了這些管理員角色之一;簡單地指定他們為檢閱者不會提升他們的特權。

## <a name="enable-the-admin-consent-workflow"></a>啟用管理員同意工作流

要啟用管理員同意工作流並選擇檢閱者，請執行以下操作：

1. 以全域管理員身份登錄到[Azure 門戶](https://portal.azure.com)。
2. 按一下左側導覽功能表頂端的 [所有服務]****。 [Azure Active Directory 擴充功能]**** 隨即開啟。
3. 在篩選器搜索框中，鍵入 **"Azure 活動目錄**"並選擇**Azure 活動目錄**項。
4. 從導覽功能表中，按一下 [企業應用程式]****。 
5. 在 **"管理**"下，選擇 **"使用者設置**"。
6. 在**管理員同意請求（預覽）** 下，設置**使用者可以要求管理員同意他們無法同意為****的應用**。

   ![建構管理員同意工作流設置](media/configure-admin-consent-workflow/admin-consent-requests-settings.png)
 
6. 設定下列設定：

   * **選擇使用者以查看管理員同意請求**。 從具有全域管理員、雲應用程式管理員和應用程式管理員角色的一組使用者中選擇此工作流的檢閱者。
   * **選定的使用者將收到請求的電子郵件通知**。 發出請求時啟用或禁用向檢閱者發送電子郵件通知。  
   * **選定的使用者將收到請求過期提醒**。 當請求即將過期時，啟用或禁用提醒電子郵件通知給檢閱者。  
   * **同意請求將在 （天） 後過期**。 指定請求保持有效的時間。

7. 選取 [儲存]****。 啟用該功能最多可能需要一個小時。

> [!NOTE]
> 您可以通過修改 **"選擇管理員同意請求檢閱者**清單"來添加或刪除此工作流的檢閱者。 請注意，此功能的當前限制是，檢閱者可以保留查看在指定為檢閱者時發出的請求的能力。

## <a name="how-users-request-admin-consent"></a>使用者如何要求管理員同意

啟用管理員同意工作流後，使用者可以要求管理員批准他們未經授權同意的應用程式。 以下步驟描述了使用者在請求審批時的體驗。 

1. 使用者嘗試登錄到應用程式。

2. 將顯示 **"批准"要求**的消息。 使用者鍵入需要訪問應用的理由，然後選擇**請求批准**。

   ![管理員同意使用者請求和理由](media/configure-admin-consent-workflow/end-user-justification.png)

3. **發送的請求**消息確認請求已提交給管理員。如果使用者發送多個請求，則僅向管理員提交第一個請求。

   ![管理員同意使用者請求和理由](media/configure-admin-consent-workflow/end-user-sent-request.png)

 4. 當使用者的請求被批准、拒絕或阻止時，使用者會收到電子郵件通知。 

## <a name="review-and-take-action-on-admin-consent-requests"></a>審核管理員同意請求並採取行動

要查看管理員同意請求並採取措施：

1. 作為管理員同意工作流的註冊檢閱者之一登錄到[Azure 門戶](https://portal.azure.com)。
2. 選擇左側導航功能表頂部**的所有服務**。 [Azure Active Directory 擴充功能]**** 隨即開啟。
3. 在篩選器搜索框中，鍵入 **"Azure 活動目錄**"並選擇**Azure 活動目錄**項。
4. 從導覽功能表中，按一下 [企業應用程式]****。
5. 在 **"活動"** 下，選擇**管理員同意請求（預覽）。**

   > [!NOTE]
   > 檢閱者將只看到在指定為檢閱者後創建的管理員請求。

1. 選擇正在請求的應用程式。
2. 查看有關請求的詳細資訊：  

   * 要查看誰在請求訪問以及原因，請選擇"**按請求"** 選項卡。
   * 要查看應用程式請求的許可權，請選擇 **"審閱許可權"和"同意**"。

8. 評估請求並採取相應操作：

   * **批准請求**。 要批准請求，請授予管理員對申請的同意。 請求獲得批准後，將通知所有要求者他們已被授予存取權限。  
   * **拒絕請求**。 要拒絕請求，必須提供將提供給所有要求者的理由。 請求被拒絕後，將通知所有要求者他們被拒絕訪問應用程式。 拒絕請求不會阻止使用者將來再次要求管理員同意應用。  
   * **阻止請求**。 要阻止請求，必須提供將提供給所有要求者的理由。 阻止請求後，將通知所有要求者他們被拒絕訪問應用程式。 阻止請求會為處於禁用狀態的租戶中的應用程式創建服務主體物件。 使用者將來將無法要求管理員同意應用程式。
 
## <a name="email-notifications"></a>電子郵件通知
 
如果配置，所有檢閱者在：

* 已創建新請求
* 請求已過期
* 請求已接近到期日期  
 
要求者將在：

* 他們提交新的訪問請求
* 他們的請求已過期
* 他們的請求被拒絕或被阻止
* 他們的請求已經獲得批准

## <a name="faq"></a>常見問題集 

**我打開此工作流，但在測試功能時，為什麼看不到新的"需要審批"提示，允許我請求訪問？**

打開該功能後，最終使用者最多可能需要 60 分鐘才能看到更新。 您可以通過在`https://graph.microsoft.com/beta/settings`API 中查看**啟用Admin同意請求**值來驗證配置是否正確生效。

**作為檢閱者，為什麼我看不到所有掛起的請求？**

檢閱者只能查看在指定為檢閱者後創建的管理員請求。 因此，如果您最近被添加為檢閱者，則看不到在分配之前創建的任何請求。

**作為檢閱者，為什麼會看到同一應用程式的多個請求？**
  
如果應用程式開發人員已將其應用配置為使用靜態和動態同意請求訪問其最終使用者的資料，您將看到兩個管理員同意請求。 一個請求表示靜態許可權，另一個請求表示動態許可權。

**作為要求者，我可以檢查請求的狀態嗎？**  

否，現在要求者只能通過電子郵件通知獲取更新。

**作為檢閱者，是否可以批准應用程式，但並非對所有人？**
 
如果您擔心授予管理員同意並允許租戶中的所有使用者使用該應用程式，我們建議您拒絕該請求。 然後，通過要求使用者分配和向應用程式分配使用者或組來限制對應用程式的訪問，從而手動授予管理員同意。 如需詳細資訊，請參閱[指派使用者和群組的方法](methods-for-assigning-users-and-groups.md)。
