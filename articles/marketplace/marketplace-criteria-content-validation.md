# <a name="azure-marketplace-listing-guidelines"></a>Azure Marketplace 列名指導方針

本檔包含 Azure Marketplace 上所列的新供應專案和服務的需求和指導方針。

所有供應專案都必須符合「列出所有供應專案的需求」一節中所列的需求。

系統會針對特定清單類型提供額外的需求和檢查清單，包括： 
* [試用版供應項目需求](#trial-offer-requirements)
* [SaaS 應用程式需求](#saas-app-requirements)
* [容器供應項目需求](#container-offer-requirements)
* [諮詢供應項目需求](#consulting-offer-requirements)

## <a name="listing-requirements-for-all-offers"></a>列出所有供應專案的需求

|  | 清單元素 | 基本需求 | 最佳需求 |
|:--- |:--- |:--- |:--- |
| 1 | 潛在客戶目的地 | 已設定潛在客戶目的地。| 一個商業夥伴（OCP）目錄具有潛在客戶目的地 CRM 資訊，也會列在 [合作夥伴解決方案] 索引標籤中。 | 
| 2 | 供應項目標題 | 簡短描述解決方案供應專案。 符合合作夥伴網站上解決方案的線上促銷。 | 包含搜尋關鍵字。 |
| 3 | 標誌 | 標誌會正確顯示。 | 標誌會正確顯示，並包含「主圖影像」，這是 Azure 入口網站中的大型格式影像。 |
| 4 |  供應項目描述 | <ul><li> 包含2-3 個段落。</li><li>一眼就能輕鬆瞭解解決方案供應專案。</li><li>沒有拼寫和文法錯誤。</li><li>是完整的，並會獲取目標物件、使用者類型，以及其價值（價值主張）的原因。</li><li>在段落敘述形式中，含有易於理解的簡短句子。</li></ul> | <ul><li> 目標產業概述（如果有相關）。</li><li>良好的樣式格式，其中每個段落標題都有一個句子或片語，會摘要說明後面的內容，並在適當時使用專案符號點，以強調主要優點。 其目標是讓讀者以易於觀看的格式快速瞭解該供應專案，而不需要閱讀長的段落。</li><li>每個段落之間會有間距。 它的閱讀方式就像汽車手冊一樣。 也就是說，它是完整的，而且不需要技術術語就能直接描述供應專案。 | 
| 5 | 類別和產業 | <ul><li>類別符合解決方案供應專案的功能。</li><li>不要延伸至方案不適用的類別。</li><li>最佳產業，或所有選取的產業（如果不是最佳產業）。 | <ul><li>最大 3個類別目錄。</li><li>最大 3個產業。 |
| 6 | 影像 | <ul><li>不需要任何映射，但如果有提供，則必須將它顯示為高解析度影像。</li><li>映射需求列在 Cloud Partner 入口網站（CPP）和此處。</li><li>螢幕擷取畫面中包含的文字會很明顯，而且影像也很清楚。 | 您可以輕鬆地瞭解解決方案供應專案。 |
| 7 | 影片 | <ul><li>不需要任何影片，但如果有提供，則必須播放，而不會發生任何錯誤。</li><li>如有提供，*除非*競爭者公司示範遷移解決方案，否則可能不會參考。 |<ul><li>在理想的情況下，長度是3分鐘或更多。</li><li>透過影片內容可輕鬆瞭解解決方案供應專案。</li><li>提供解決方案功能的示範。 |
| 8 | 列出狀態（呼叫動作） | <ul><li>必須標示為下列其中一種類型： <ul><li>*聯絡我*</li><li>*試用*/*取得試用版現在*/*開始試用試用產品*/ * *</li><li>*立即購買*以立即*取得*/</li></ul></ul> | 客戶可以隨時瞭解接下來的步驟： <ol><li>試試試用。</li><li>立即購買。</li><li>透過電子郵件或電話號碼聯絡，以安排概念證明（POC）、評量或簡報。</li></ol> | 
| 9 | 解決方案定價 | 必須有解決方案定價索引標籤/詳細資料，而且定價必須以合作夥伴解決方案供應專案的當地貨幣為單位。 | 有多個計費選項應該提供給客戶選擇。 |    
| 10 | 深入了解 | 底部的連結（在 [描述] 底下，而不是左邊的 Azure Marketplace 連結）會導致解決方案的詳細資訊，而且可公開使用並正確顯示。 |    連結至特定專案（例如，合作夥伴網站上的規格頁面），而不只是合作夥伴首頁。 | 
| 11 | 解決方案支援與協助 | 提供至少下列一個的連結： <ul><li>電話號碼</li><li>電子郵件支援</li><li>聊天代理程式</li><li>社區論壇 |<ul><li>列出所有支援方法。</li><li>在*試用版或試用產品*期間免費提供*Test Drive*付費支援。 | 
| 12 | 法律 | 原則或條款可透過公用 URL 取得。 |  |

## <a name="saas-app-requirements"></a>SaaS 應用程式需求

|  | 清單元素 | 基本需求 | 最佳需求 | 
|:--- |:--- |:--- |:--- |

| 2 | 技術資訊：設定 |<ul><li>針對軟體即服務（SaaS）應用程式，選擇您只想要列出您的應用程式，或讓客戶透過 Azure 購買您的應用程式。</li><li>在供應專案的 [取得] 按鈕上選取您想要的文字： [*免費*]、[*免費試用*] 或 [*聯絡我*]。</li><li>在快顯視窗中，如果您的應用程式利用技術，請只選取一個適用的產品： Cortana 智慧、Power BI 解決方案範本或 Power Apps。 |  |
| 3 | 試用產品 | 選取 *[是] 或 [* *否*] | 客戶可以隨時瞭解接下來的步驟： <ol><li>試試試用。</li><li>立即購買。</li><li>透過電子郵件或電話號碼聯絡，以安排 POC、評量或簡報。</li></ol> |
| 4 | 店面詳細資料：供應專案摘要 | 出現在您應用程式的搜尋頁面上，最多可有100個字元。 | |
| 5 | 店面詳細資料：產業 | 產業（最多2個）：選取您的應用程式最適合且適用的產業。 | |
| 6 | 供應項目描述 | <ul><li>允許使用簡單的 HTML， \<包括 p> \<、em> \<、ul> \<、li> \<、ol> 和標頭標記。 最大 3000個字元。</li><li>包含2-3 個段落。</li><li>一眼就能輕鬆瞭解解決方案供應專案。</li><li>「描述」是完整的，並可讓您瞭解目標物件、使用者類型，以及其價值（價值主張）。</li><li>是以段落敘述形式呈現，其中包含容易瞭解的簡短句子。 | <ul><li>概述目標產業（如果相關）。</li><li>良好的樣式格式，其中每個段落標題都有一個句子或片語，會摘要說明後面的內容，並在適當時使用專案符號點，以強調主要優點。 其目標是讓讀者以易於觀看的格式快速瞭解該供應專案，而不需要閱讀段落。</li><li>每個段落之間會有間距。 它的閱讀方式就像汽車手冊一樣。 也就是說，它是完整的，而且不需要技術術語就能直接描述供應專案。 | 
| 7 | 行銷成品 | 標誌會正確顯示。 | <ul><li>標誌包含「主圖影像」，這是 Azure 入口網站中的大型格式影像。</li><li>標誌：小型（48 &#215; 48 圖元）和大型（216 &#215; 216 圖元）是必要的。</li><li>主[圖映射需求](https://docs.microsoft.com/azure/marketplace/partner-center-portal/create-new-azure-apps-offer)。</li><li>螢幕擷取畫面（最多5個）： CPP 需要解析度為 1280 &#215; 720 圖元的 PNG 影像。 |
| 8 | 類別和產業 | <ul><li>類別符合解決方案提供的功能。</li><li>在快顯視窗中至少選取一個專案。</li><li>不要延伸至方案不適用的類別。</li><li>最佳產業，或所有選取的產業（如果不是最佳產業）。 | 最大 已選取3個分類（如果適用的話）。 |
| 9 | 潛在客戶管理 | 選取將儲存您潛在客戶的系統。 請參閱[取得客戶領導](./partner-center-portal/commercial-marketplace-get-customer-leads.md)以連接您的 CRM 系統。 |  |
| 10 | 連絡人：解決方案支援與協助 | <ul><li>工程連絡人名稱：您的應用程式的工程連絡人名稱。 這名連絡人將收到來自 Microsoft 的技術通訊。</li><li>工程連絡人電子郵件：應用程式的工程連絡人電子郵件地址。</li><li>工程連絡人電話：工程連絡人的電話號碼。 支援[ISO 電話號碼標記法](https://en.wikipedia.org/wiki/E.123)。</li><li>支援連絡人名稱：您的應用程式的支援連絡人名稱。 這名連絡人將收到來自 Microsoft 的支援相關通訊。</li><li>支援連絡人電子郵件：您應用程式支援連絡人的電子郵件地址。</li><li>支援連絡人電話：支援連絡人的電話號碼。 支援[ISO 電話號碼標記法](https://en.wikipedia.org/wiki/E.123)。</li><li>支援 URL：支援頁面的 URL。 | <ul><li>列出所有支援方法。</li><li>在*試用版或試用產品*期間提供*Test Drive*免費的付費支援。 |
| 11 | 法律 |<ul><li>隱私權原則 URL：在 CPP 的 [隱私權原則 URL] 欄位中，您的應用程式隱私權原則的 URL。</li><li>使用規定：應用程式的使用條款。 客戶必須先接受這些條款，才能試用您的應用程式。 |    原則或條款可透過公用 URL 網站取得。 |

|:--- |:--- |:--- |:--- |

| 3 | Markdown 格式  | 所有供應專案都必須使用 Markdown 格式，才能在它轉換成 HTML 時正確轉譯。  |  |
| 4 | 類別和產業 | 類別不相關。 |  | 
| 5 | 列出狀態（呼叫動作） | 自動列為 [*連絡人*]。 |  | 
| 6 | 解決方案支援與協助  | 不需要支援和協助。  | | 
| 7 | 隱私權原則和使用條款  |  不需要原則或詞彙。  | | 
| 8 | 服務類型 | 請確定服務類型符合標題。 | |
| 9 | 專長認證 | 至少必須具備下列其中一項的專長認證：<ul><li>應用程式開發</li><li>應用程式整合</li><li>應用程式生命週期管理</li><li>雲端平台<li>資料分析</li><li>資料中心</li><li>資料平台</li><li>DevOps |

| 11 | 國家/地區 | 請確定國家/地區符合選取的貨幣。 | |
| 12 | 深入了解  | <ul><li>底部的連結（在 [描述] 底下，而不是左邊的 Azure Marketplace 連結）會導致解決方案的詳細資訊，而且可公開使用並正確顯示。</li><li>連結必須有「易記名稱」，而且不會顯示為任何下載的檔案名。 | |
