# <a name="azure-marketplace-listing-guidelines"></a>Azure Marketplace 列名指導方針

本文說明在 Azure Marketplace 上列出新供應項目和服務的需求與指導方針。 所有供應專案都必須符合下列列出的 [需求](#listing-requirements-for-all-offers) 。 使用右側的連結，流覽至特定清單類型的其他需求和檢查清單。

## <a name="listing-requirements-for-all-offers"></a>所有供應項目的列出需求

| 否。 | 列出元素 | 基本需求 | 最佳需求 |
|:--- |:--- |:--- |:--- |
| 1 | 潛在客戶目的地 | 已設定潛在客戶目的地。| One Commercial Partner (OCP) 目錄具有潛在客戶目的地 CRM 資訊，此資訊也列在 [合作夥伴解決方案] 索引標籤中。 |
| 2 | 供應項目標題 | 簡短描述解決方案供應項目。 符合合作夥伴網站上解決方案的線上促銷。 | 包含搜尋關鍵字。 |
| 3 | 標誌 | 標誌會正確顯示。  | 標誌會正確顯示。  |
| 4 |  供應項目描述 | <ul><li> 包含 2 到 3 個段落。</li><li>解決方案供應項目可一目了然。</li><li>沒有拼寫和文法錯誤。</li><li>詳盡且能緊扣目標對象、使用者類型和為何有價值 (價值主張)。</li><li>使用易於理解的簡短句子分段敘述。</li></ul> | <ul><li> 有概述目標產業 (如果有相關)。</li><li>良好的樣式格式，每個段落標題都有一個句子或片語概述後續的內容，且使用項目符號 (如果適當時) 以強調重要的優點。 目標是讓閱讀者能以易於檢視的格式一眼就了解供應項目，而不需要閱讀長篇大論。</li><li>每個段落之間有所間隔。 版面看起來就像汽車銷售小冊。 也就是說，內容詳盡，而且簡單描述供應項目，沒有技術方面的術語。 | 
| 5 | 類別和產業 | <ul><li>類別和子類別必須符合供應專案功能 </li><li>請勿選取不符合您供應專案功能的類別/子類別。 </li></ul> | <ul><li>選取最多兩個類別，包括主要和次要類別 (選擇性) 。</li><li>針對每個主要和/或次要類別，最多選取兩個子類別。 如果未選取任何子類別，則您的供應專案仍可在選取的類別中找到。</li></ul> |
| 6 | 影像 | <ul><li>映射需求列在合作夥伴中心中。</li><li>螢幕擷取畫面中包含的文字很清晰，影像也很清楚。 | 解決方案供應項目可一目了然。 |
| 7 | 影片 | <ul><li>不需要任何影片，但如果有提供，則必須能正常播放，不會有任何錯誤。</li><li>如果有提供，則不得提及競爭對手公司，*除非* 是示範移轉解決方案。 |<ul><li>理想的長度是 3 分鐘以上。</li><li>可透過影片內容輕鬆了解解決方案供應項目。</li><li>提供解決方案功能的示範。 |
| 8 | 列出狀態 (清單選項)  | <ul><li>必須標示為下列其中一種類型： <ul><li>*與我連絡*</li><li>*試用版*/*立即取得試用版*/*開始試用*/*試用產品*</li><li>*立即購買*/*立即取得*</li></ul></ul> | 客戶可以輕易地了解有哪些後續步驟： <ol><li>嘗試試用版。</li><li>立即購買。</li><li>透過電子郵件或電話號碼連絡以安排概念證明 (POC)、評估或簡報。</li></ol> | 
| 9 | 解決方案定價 | 必須有解決方案定價索引標籤/詳細資料，且必須以合作夥伴解決方案供應項目的當地貨幣來定價。 | 應該提供多個計費選項，並有層級定價給客戶選擇。 |
| 10 | 深入了解 | 在底部提供有關解決方案詳細資訊的連結 (在描述的底下，而非左側的 Azure Marketplace 連結)，且連結為公開可用並正確顯示。 |    有特定項目 (例如，合作夥伴網站上的規格頁面) 的連結，而不是只有合作夥伴首頁的連結。 | 
| 11 | 解決方案支援和協助 | 提供至少下列一個的連結： <ul><li>電話號碼</li><li>電子郵件支援</li><li>聊天代理程式</li><li>社群論壇 |<ul><li>列出所有支援方法。</li><li>在 *試用版* 或 *試用產品* 使用期間免費提供付費支援。 |
| 12 | 法律 | 可透過公開 URL 取得原則或條款。 |  |

## <a name="saas-application-requirements"></a>SaaS 應用程式需求

| 否。 | 列出元素 | 基本需求 | 最佳需求 | 
|:--- |:--- |:--- |:--- |
| 1 | 供應項目標題 |<ul><li>只能包含小寫字母、英數字元、連字號或底線。 發佈後就不能修改標題。</li><li>描述解決方案供應項目。</li><li>符合合作夥伴網站上解決方案的線上促銷。 | 包含搜尋關鍵字。 |
| 2 | 技術資訊：組態 |<ul><li>針對軟體即服務 (SaaS) 應用程式，選擇您只想要列出您的應用程式，還是要讓客戶可以透過 Azure 購買您的應用程式。</li><li>選取要在供應項目的 [取得] 按鈕上顯示的文字：*免費*、*免費試用* 或 *與我連絡*。</li><li>在快顯視窗中只選取一個適用的產品 (如果您的應用程式利用該技術的話)：Cortana Intelligence、Power BI 解決方案範本或 Power Apps。 |  |
| 3 | 試用產品 | 選取 [是] 或 [否] | 客戶可以輕易地了解有哪些後續步驟： <ol><li>嘗試試用版。</li><li>立即購買。</li><li>透過電子郵件或電話號碼連絡以安排概念證明 (POC)、評估或簡報。</li></ol> |
| 4 | 線上商店詳細資料：優惠摘要 | 出現在您應用程式的搜尋頁面上，最多可有 100 個字元。 | |
| 5 | 線上商店詳細資料：產業 | 產業 (最多 2 個)：選取您應用程式最吻合且最適合的產業。 | |
| 6 | 供應項目描述 | <ul><li>允許使用簡單的 HTML，包括、、、 \<p> \<em> \<ul> \<li> 、 \<ol> 和標頭標記。 最大 3000 個字元。</li><li>包含 2 到 3 個段落。</li><li>解決方案供應項目可一目了然。</li><li>描述詳盡且能緊扣目標對象、使用者類型和為何有價值 (價值主張)。</li><li>使用易於理解的簡短句子分段敘述。 | <ul><li>有概述目標產業 (如果有相關)。</li><li>良好的樣式格式，每個段落標題都有一個句子或片語概述後續的內容，且使用項目符號 (如果適當時) 以強調重要的優點。 目標是讓閱讀者能以易於檢視的格式一眼就了解供應項目，而不需要閱讀長篇大論。</li><li>每個段落之間有所間隔。 版面看起來就像汽車銷售小冊。 也就是說，內容詳盡，而且簡單描述供應項目，沒有技術方面的術語。 | 
| 7 | 行銷成品 | 正確顯示標誌。 | <ul><li>標誌：小規模 (48 x 48 圖元、選擇性) 、中 (90 x 90 px、選擇性) ，以及從 216 x 216 至 350 x 350 圖元的大型 (，需要) 。</li><li> (最大值 5) 的螢幕擷取畫面：需要。解析度為1280x720 圖元的 PNG 影像。 |
| 8 | 類別和產業 | <ul><li>類別符合解決方案供應項目的功能。</li><li>在快顯視窗中至少選取一個項目。</li><li>請勿選取方案無法容納的類別。</li><li>最適合的產業，或所有選取的產業 (如果沒有最適合的產業)。 | 最大 選取 3 個類別 (如果適用的話)。 |
| 9 | 潛在客戶管理 | 選取將儲存您潛在客戶的系統。 請參閱[取得潛在客戶](./partner-center-portal/commercial-marketplace-get-customer-leads.md)以連接您的 CRM 系統。 |  |
| 10 | 連絡人：解決方案支援和協助 | <ul><li>工程連絡人姓名：應用程式的工程連絡人姓名。 這名連絡人將收到來自 Microsoft 的技術通訊。</li><li>工程連絡人電子郵件：應用程式的工程連絡人電子郵件地址。</li><li>工程連絡人電話：工程連絡人的電話號碼。 支援 [ISO 電話號碼標記法](https://en.wikipedia.org/wiki/E.123)。</li><li>支援連絡人姓名：應用程式的支援連絡人姓名。 這名連絡人將收到來自 Microsoft 的支援相關通訊。</li><li>支援連絡人電子郵件：應用程式的支援連絡人電子郵件地址。</li><li>支援連絡人電話：支援連絡人的電話號碼。 支援 [ISO 電話號碼標記法](https://en.wikipedia.org/wiki/E.123)。</li><li>支援 URL：支援網頁的 URL。 | <ul><li>列出所有支援方法。</li><li>在 *試用版* 或 *試用產品* 使用期間免費提供付費支援。 |
| 11 | 法律 |<ul><li>隱私權原則 URL：應用程式的隱私權原則 URL (在 CPP 中的 [隱私權原則 URL] 欄位)。</li><li>使用規定：應用程式的使用規定。 客戶必須先接受這些條款，才能試用您的應用程式。 |    可透過公開 URL 網站取得原則或條款。 |

## <a name="consulting-offer-requirements"></a>諮詢供應項目需求

|:--- |:--- |:--- |:--- |

| 2 | 供應項目描述  |<ul><li>確定有正確使用 [Microsoft 產品名稱](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/en-us.aspx)。</li><li>標示為「價格：估計」的供應項目必須在頂端或底部註記以說明各種變化 (到用戶端出差、要遷移的伺服器數量等等)。</li><li>每個供應項目都有描述需求，如下所示： <ul><li>簡報需要至少四個或五個項目符號，並提供所涵蓋主題的相關資訊。</li><li>所有研討會都需要有議程。</li><li>所有議程都必須以日或週為單位來細分，視研討會的持續時間而定。</li><li>評估、POC、非訓練型研討會和實作供應項目需要有可交付成果。</li><li>訓練研討會不需要可交付成果，但需要要更詳細的議程與所會涵蓋的主題。</li></ul></ul> | <ul><li>任何供應項目都有議程和可交付成果。</li><li>供應項目頂端區段包含一個段落，含有提供服務的公司相關內容。</li><li>供應項目在頂端區段納入服務本身價值的相關段落。 |
| 3 | Markdown 格式  | 所有供應項目都必須使用 Markdown 格式，如此在轉換為 HTML 時才會正確呈現供應項目。  |  |
| 4 | 類別和產業 | 類別無相關性。 |  | 
| 5 | 列出狀態 (清單選項)  | 自動列為 *與我連絡*。 |  | 
| 6 | 解決方案支援和協助  | 不需要支援和協助。  | | 
| 7 | 提供隱私權原則和使用規定  |  不需要原則或條款。  | | 
| 8 | 服務類型 | 請確定服務類型符合標題。 | |
| 9 | 專長認證 | 至少必須具備下列其中一個領域的專長認證：<ul><li>應用程式開發</li><li>應用程式整合</li><li>應用程式生命週期管理</li><li>雲端平台<li>資料分析</li><li>資料中心</li><li>資料平台</li><li>DevOps |
| 10 | Products | 必須是 Azure 產品。 | |
| 11 | 國家/地區 | 請確定國家/地區符合選取的貨幣。 | |
| 12 | 深入了解  | <ul><li>在底部提供有關解決方案詳細資訊的連結 (在描述的底下，而非左側的 Azure Marketplace 連結)，且連結為公開可用並正確顯示。</li><li>連結必須有「自訂名稱」，且不會顯示為任何下載的檔案名稱。 | |
