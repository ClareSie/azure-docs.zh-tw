# <a name="determining-application-type"></a>判斷應用程式類型

在超大規模資料庫上執行有效率的查詢 (Citus) 伺服器群組需要將資料表正確散發到伺服器上。 建議的散發會因應用程式類型及其查詢模式而有所不同。

在超大規模資料庫 (Citus) 上，有廣泛的應用程式可以順利運作。 資料模型化的第一個步驟是識別哪些專案與您的應用程式非常類似。

## <a name="at-a-glance"></a>一覽

| 多租使用者應用程式                                 | 即時應用程式                                |
|-----------------------------------------------------------|-------------------------------------------------------|
| 有時候架構中的數十或數百個數據表          | 少數資料表                                |
| 一次與一個租使用者相關的查詢 (公司/商店)  | 具有匯總的相對簡單分析查詢 |
| 為 web 用戶端提供服務的 OLTP 工作負載                    | 大部分不可變數據的大量內嵌磁片區           |
| 提供每個租使用者分析查詢的 OLAP 工作負載   | 經常在事件的大型資料表周圍            |

## <a name="examples-and-characteristics"></a>範例和特性

**多租使用者應用程式**

> 這些通常是為其他公司、帳戶或組織提供服務的 SaaS 應用程式。 大部分的 SaaS 應用程式本質上都是關聯式的。 它們具有用來在節點間散發資料的自然維度：僅依租使用者 \_ 識別碼分區。
>
> 超大規模資料庫 (Citus) 可讓您將資料庫相應放大至數百萬個租使用者，而不需要重新架構您的應用程式。 您可以保留所需的關聯式語義，例如聯結、外鍵條件約束、交易、ACID 和一致性。
>
> -   **範例**：主控商店的網站-其他企業的正面，例如數位行銷解決方案或銷售自動化工具。
> -   **特性**：與單一租使用者相關的查詢，而不是在租使用者之間聯結資訊。 這包括用於服務 web 用戶端的 OLTP 工作負載，以及提供每個租使用者分析查詢的 OLAP 工作負載。 在您的資料庫架構中擁有數十個或數百個數據表也是多租使用者資料模型的指標。
>
> 使用超大規模資料庫 (Citus) 調整多租使用者應用程式，也需要對應用程式代碼進行最低限度的變更。 我們支援熱門的架構，例如 Ruby on Rails 和 Django。

**即時分析**

> 需要大量平行處理的應用程式，協調數百個核心以快速產生數值、統計或計算查詢。  藉由在多個節點之間分區化和平行處理 SQL 查詢，超大規模資料庫 (Citus) 讓您能夠在一秒鐘的時間內，對數十億筆記錄執行即時查詢。
>
> 即時分析資料模型中的資料表通常是由使用者 \_ 識別碼、主機 \_ 識別碼或裝置識別碼等資料行所散發 \_ 。
>
> -   **範例**：客戶面向的分析儀表板，需要二秒的回應時間。
> -   **特性**：較少的資料表，通常是在裝置、網站或使用者事件的大型資料表中心，而且大部分的不可變數據都需要大量的內嵌量。 相對簡單的 (，但需要大量運算的) 分析查詢牽涉到數個匯總和群組 BYs。

如果您的情況類似上述任一案例，則下一個步驟是決定如何在伺服器群組中分區您的資料。 「資料庫管理員」的 \' 散發資料行選擇必須符合一般查詢的存取模式，以確保效能。
