# <a name="azure-marketplace-listing-guidelines"></a>Azure Marketplace 列名指導方針

本文件包含適用於檢閱者針對列名至 Azure Marketplace 之供應項目和服務的需求指導方針與檢查清單。

所有供應項目都必須符合以下所示的[所有列名需求](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#all-listing-requirements)。 針對特定列名類型，會提供其他需求和檢查清單，包括：附加[試用版](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#trial-offer-requirements)的供應項目、[SaaS 應用程式](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#saas-app-requirements)、[容器](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#container-offer-requirements)和[諮詢供應項目](https://docs.microsoft.com/azure/marketplace/marketplace-criteria-content-validation#consulting-offer-requirements)。


## <a name="all-listing-requirements"></a>所有列名需求

|  | 列名項目 | 基本需求 | 最佳需求 |
|:--- |:--- |:--- |:--- |
| 1 | 潛在客戶目的地 | 已設定潛在客戶目的地| 具有潛在客戶目的地 CRM 資訊的 OCP 目錄也列在合作夥伴解決方案索引標籤中 | 
| 2 | 供應項目標題 | 描述解決方案供應項目。 在合作夥伴的網站上符合解決方案的線上推廣 | 包含搜尋關鍵字 |
| 3 | 標誌 | 正確地顯示標誌 | 徽標顯示正確,並在 Azure 門戶中包含 [英雄圖像,這是大格式圖像 |
| 4 |  供應項目描述 | <br>• 2-3 段 <br>• 解決方案產品一目了然,很容易理解 <br>* 提供描述是沒有拼字與語法錯誤 <br>• 優惠描述是全面的和捕獲的:目標受眾,用戶類型,為什麼它的價值(價值道具) <br>* 報價描述是段落敘述形式與短句,易於理解 | <br>• 目標產業概述(如果相關) <br>• 良好的樣式格式,每個段落標題都有一個句子或短語,彙總以下內容,並包括要點(適當時),以強調關鍵優勢。 此處的目標是讓閱讀者能以易於檢視的格式一眼就了解供應項目，而不需要閱讀段落內容。 <br>• 每個段落之間都有間距,讀起來就像汽車宣傳冊一樣。 意思並不是要描述技術功能或程式碼術語，而是以簡單的字詞和完整的描述來說明供應項目。 | 
| 5 | 類別和產業 | <br>• 類別符合解決方案提供功能 <br>• 不要擴展到解決方案不適合的類別 <br>• 最佳行業或所選所有行業(如果不是最佳行業) | <br>• 最大。 3 個類別 <br>• 最大。 3 個產業 |
| 6 | 影像 | <br>• 無需圖像,但如果提供,則必須顯示高解析度圖像。 <br>• 影像要求列在合作夥伴中心,也在這裡 <br>• 螢幕截圖中包含的文字清晰可辨,影像清晰 | 解決方案供應項目是一目了然的 |
| 7 | 影片 | <br>• 無需視頻,但如果提供,則必須播放沒有任何錯誤。 <br>• 如果提供,不得指競爭對手公司,除非演示遷移解決方案 |<br>• 理想情況下 3 分鐘或更長時間 <br>• 解決方案提供透過視訊內容輕鬆理解的解決方案 <br>• 解決方案功能展示 |
| 8 | 列名狀態 (行動呼籲) | <br>必須標記以下類型之一: <ul>o 與我聯絡</ul><ul>o 試用版/立即取得試用版/開始試用/試用產品</ul><ul>o 立即購買/立即購買</ul> | 客戶可以輕易地了解有哪些可用的後續步驟： <ul>1. 試用試用版</ul> <ul>2. 立即購買</ul><ul>3. 通過電子郵件或電話號碼聯繫,安排 PoC、評估或簡報。</ul> | 
| 9 | 解決方案價格 | 必須有解決方案價格索引標籤/詳細資料，且以合作夥伴解決方案供應項目的當地貨幣為單位 | 使用定價層提供多種計費選項供客戶選擇 |    
| 10 | 深入了解 | 底部的連結(在描述下,而不是左側的市場連結)導致有關解決方案的詳細資訊,並且公開可用並正確顯示 |    指向特定項目的連結(例如,合作夥伴網站上的規格頁面),而不僅僅是合作夥伴主頁網站 | 
| 11 | 解決方案支援與說明 | 提供至少下列一個的連結： <br>O 電話號碼 <br>電子郵件支援 <br>o 聊天代理 <br>o 社區論壇 |<br>• 列出的所有支援方法:電話&電子郵件支援。 <br>• 在試駕期間免費提供付費支援 | 
| 12 | 法律 | 透過公開 URL 提供原則或條款 |  |

## <a name="consulting-offer-requirements"></a>諮詢供應項目需求

| 2 | 供應項目描述  |<br>• 確保正確使用[微軟產品名稱](https://www.microsoft.com/en-us/legal/intellectualproperty/trademarks/en-us.aspx) <br>• 標記為價格的報價:估計必須在優惠的頂部或底部有一個註釋來解釋可變性(前往用戶端、遷移的伺服器數量等) <br>• 每種報價類型都有如下描述要求: <ul><br>o 簡報中針對簡報涵蓋的主題資訊至少需要 4 到 5 個項目符號來說明 </ul> <ul><br>o 所有工作坊都需要日程表 </ul><ul><br>o 所有日程表都必須細分以日或週為單位，視工作坊的持續時間而定 </ul><ul><br>o 評量、POC、非訓練型工作坊，實作的供應項目則需要交付項目 </ul><ul><br>o培訓研討會不需要可交付成果,但它們需要更詳細的議程,其中將涵蓋主題。 | <br>• 任何報價都有議程和可交付成果 <br>* 優惠包括一段,其中包含有關在頂部部分提供服務的公司上下文 <br>• 優惠包括一段關於服務本身作為頂部部分的價值 |
| 3 | Markdown 格式  | 所有供應項目都必須使用 Markdown 格式，如此在轉換為 HTML 時才會正確轉譯供應項目  |  |
| 4 | 類別和產業 | 不相關的類別 |  | 
| 5 | 列名狀態 (CTA) | 自動列為與我連絡 |  | 
| 6 | 解決方案支援與說明  | 支援與說明為非必要  | | 
| 7 | 提供隱私權原則及使用條款  |  原則或條款為非必要  | | 
| 8 | 服務類型 | 確定服務類型與標題相符 | |
| 9 | 專長認證 |<br>• 必須至少為以下一項:<br>• 應用程式開發 <br>• 應用程式整合 <br>• 應用程式生命週期管理 <br>• 雲平臺 <br>• 資料分析 <br>• 資料中心 <br>• 資料平台 <br>• DevOps |
| 10 | Products | 必須是 Azure | |
| 11 | 國家/區域 | 確定國家和地區與選擇的貨幣相符 | |
| 12 | 深入了解  | <br>• 底部的連結(在描述下,而不是左側的市場連結)導致有關解決方案的詳細資訊,並公開提供並正確顯示。 <br>• 連結必須具有「友好名稱」,不得顯示為任何下載的檔名 | |
